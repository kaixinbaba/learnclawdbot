# GitHub 統合とは？（開発者の司令塔）

> GitHub 統合は、**DevOps 支援ツール**であり、**会話を通じてコードリポジトリと Issue を管理**するために使用され、**プログラマーや技術リーダー**が【Web 版 GitHub を開くのが不便なシーン】において【プロジェクトの進捗をリアルタイムで追跡し、緊急のバグを処理する】のを支援します。

---

## 一言で言うと

- **解決する問題**：Issue のステータスを確認したり、コードの一部を探したりするためだけに、IDE とブラウザを頻繁に行き来したくない。
- **コアバリュー**：リアルタイム通知、セマンティックコード検索、自然言語による Issue 操作。
- **典型的な使用例**：アラートを受け取った後、最近誰がコードをマージしたかをすぐに確認する、スマホで Issue コメントに返信する、AI に特定の PR が何を変更したか説明させる。

---

## 1. GitHub 統合とは？

GitHub 統合は、本質的に**コードリポジトリの隣に駐在する AI アシスタント**です。

これを**24時間オンラインのテックリード**として理解できます：
- すべてのアクティビティ（Star、Fork、Push、PR）を監視しています。
- 「最近あのログインバグは修正された？」と聞くと、直接 Issue のステータスを調べて教えてくれます。

---

## 2. 何ができる？

通常、GitHub 統合は以下の用途に使用できます：

1.  **スマート巡回**（一言で）：「ここ1週間で未解決の P0 級バグをリストアップして」と聞くと、リンク付きでリストアップします。
2.  **PR 要約**：同僚が50ファイルの PR を提出したが、見る時間がない？AI に Diff を読ませて、数百文字の要約を書いてもらいましょう。
3.  **コード検索**：「Redis ロックはどこで使ってる？」と聞くと、リポジトリを走査して具体的なファイルと行番号を見つけます。

実際の製品では、一般的な使用法は以下の通りです：
-   **Star 監視**：プロジェクトが新しい Star を獲得したときに通知します（初期ユーザーを探している独立開発者に適しています）。
-   **Issue 自動返信**：Webhook を通じて、ユーザーが Issue を提出した際、再現手順が不足していないか AI が自動判断し、リマインドを返信します。
-   **リリースノート生成**：最近の Commit 記録に基づいて、自動的に Release Note のドラフトを生成します。

---

## 3. 使い方は？

一般的に、GitHub 統合の使用は 3 つのステップに分かれます：

1.  **ステップ 1：PAT の生成**
    GitHub の `Settings` -> `Developer settings` -> `Personal access tokens` でトークンを生成します。より細かい権限制御のために `Fine-grained tokens` を選択することをお勧めします。

2.  **ステップ 2：Scope の設定**
    トークンに `repo`（リポジトリの読み書き）、`user`（ユーザー情報の読み取り）などの必要な権限がチェックされていることを確認してください。

3.  **ステップ 3：OpenClaw のバインド**
    OpenClaw 管理画面にトークンを入力します。

初めて使用する場合：
-   **【Issue 確認】から始めることをお勧めします**：「私にアサインされた issue は何？」と聞いてみてください。
-   **最初から【書き込み権限】を有効にしないでください**：安全のため、初期は読み取り専用（Read-only）権限のみを与え、AI によるコードの誤操作を防ぐことをお勧めします。

---

## 4. なぜこのように GitHub 統合を使用するのか？

AI による GitHub 操作支援を推奨する主な理由は以下の通りです：

-   **理由 1：クロスプラットフォーム**：スマホでの GitHub App の体験がいまいちな場合、チャットソフトで直接 Bot に聞くほうが速いです。
-   **理由 2：コンテキスト理解**：GitHub 検索はキーワードしか検索できませんが、AI はセマンティクス（「ログインロジックを処理するコードを探して」など）を検索できます。
-   **理由 3：情報の集約**：AI は Commit メッセージ、Issue の議論、PR の説明を組み合わせて、より包括的な分析を提供できます。

純粋に手動管理を選択した場合：
-   数百の Issue に直面したとき、優先度の高いものを素早くフィルタリングするのは困難です。
-   大規模な PR をレビューするとき、大量の Diff の中で迷子になりやすいです。

---

## 5. 誰に適している？誰に適していない？

**適している人:**
-   **オープンソースメンテナ**：毎日大量の Issue と PR を処理する人。
-   **テックリーダー**：プロジェクト全体の進捗と品質を管理する必要がある人。
-   **独立開発者**：一人でチーム分の仕事をこなし、補助する自動化ツールが必要な人。

**あまり適していない人:**
-   **機密プロジェクト**：会社の規定でコードの持ち出しやサードパーティによる読み取りが厳禁されている場合。
-   **非コードプロジェクト**：GitHub をドキュメント保存だけに使っている場合、Notion 統合の方が適しているかもしれません。
