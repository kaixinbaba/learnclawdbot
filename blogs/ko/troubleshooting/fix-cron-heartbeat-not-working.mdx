# 수정: OpenClaw에서 Cron 작업 및 Heartbeat가 작동하지 않음

OpenClaw 사용자가 직면하는 가장 일반적인 문제 중 하나는 예약된 작업이 실행되지 않는 것입니다. cron 작업이나 heartbeat를 설정했지만 아무 일도 일어나지 않습니다. 이 가이드는 가장 빈번한 원인과 수정 방법을 다룹니다.

## 빠른 진단

깊이 파고들기 전에 다음 확인을 실행하세요:

```bash
# Check cron status
openclaw cron status

# List all scheduled jobs
openclaw cron list

# Check gateway logs for scheduler errors
tail -100 ~/.openclaw/logs/gateway.log | grep -i "cron\|heartbeat\|scheduler"
```

## 일반적인 문제 #1: Cron 작업이 전혀 실행되지 않음

### 증상
- 작업이 `enabled: true`로 표시되지만 실행되지 않음
- 로그에 오류가 없고 작업이 조용히 실행되지 않음
- `openclaw cron list`에 작업이 표시되지만 `lastRun`이 항상 null

### 원인: Gateway가 실행 중이지 않거나 재시작됨

Cron 작업은 Gateway가 실행 중일 때만 실행됩니다. 다음의 경우:
- 터미널을 닫음
- 컴퓨터를 재시작함  
- `openclaw gateway stop`을 실행함

cron 작업도 중지됩니다.

### 수정 방법

```bash
# Check if gateway is running
openclaw gateway status

# If not running, start it
openclaw gateway start

# Or restart to pick up config changes
openclaw gateway restart
```

**전문가 팁:** `openclaw gateway start --daemon`을 사용하여 백그라운드에서 실행하거나 시스템 서비스로 설정하세요.

## 일반적인 문제 #2: Heartbeat가 실행되지 않음

### 증상
- Heartbeat 폴링이 구성되었지만 에이전트가 응답하지 않음
- `HEARTBEAT_OK` 또는 능동적인 메시지가 없음
- 직접 메시지를 보내면 정상적으로 작동함

### 원인 A: 잘못된 wakeMode

가장 일반적인 원인은 `wakeMode: "next-heartbeat"`의 의미를 이해하지 못하고 사용하는 것입니다.

```json
// ❌ 이것은 다음 heartbeat 주기를 기다리며, 즉시 실행되지 않습니다
{
  "schedule": { "kind": "every", "everyMs": 1800000 },
  "payload": { "kind": "systemEvent", "text": "Check inbox" },
  "wakeMode": "next-heartbeat"
}
```

### 수정 방법

즉시 실행하려면 `wakeMode: "now"`를 사용하거나 생략하세요:

```json
// ✅ 예약 시 즉시 실행
{
  "schedule": { "kind": "every", "everyMs": 1800000 },
  "payload": { "kind": "systemEvent", "text": "Check inbox" }
}
```

### 원인 B: 빈 HEARTBEAT.md

`HEARTBEAT.md` 파일이 비어 있거나 주석만 포함된 경우, 에이전트는 올바르게 `HEARTBEAT_OK`를 반환하고 아무것도 하지 않습니다.

### 수정 방법

`HEARTBEAT.md`에 실제 작업을 추가하세요:

```markdown
# HEARTBEAT.md

- Check for unread emails (if not checked in last 2 hours)
- Review calendar for upcoming events in next 24h
- Check project status if any tasks are overdue
```

## 일반적인 문제 #3: 일회성 작업이 무한 반복됨

### 증상
- `schedule.kind: "at"`으로 일회성 작업을 생성함
- 작업이 한 번이 아니라 반복적으로 실행됨
- 상태가 "skipped"로 표시되지만 작업이 계속 트리거됨

### 원인: Skipped 상태의 deleteAfterRun 버그

일부 버전에서는 `deleteAfterRun: true`가 있는 일회성 작업이 상태가 "skipped"일 때 제대로 삭제되지 않습니다.

### 수정 방법

**옵션 1:** 실행 후 수동으로 작업 삭제:
```bash
openclaw cron remove --id <job-id>
```

**옵션 2:** 우회 스케줄 사용:
```json
{
  "schedule": { 
    "kind": "at", 
    "at": "2026-02-10T15:00:00Z"
  },
  "deleteAfterRun": true,
  "payload": { "kind": "systemEvent", "text": "Reminder: Call mom" }
}
```

**옵션 3:** 이것이 수정된 최신 OpenClaw 버전으로 업데이트.

## 일반적인 문제 #4: 격리된 세션 작업이 알림하지 않음

### 증상
- `sessionTarget: "isolated"` 작업이 실행되지만 출력을 볼 수 없음
- `openclaw cron runs`에서 작업이 완료로 표시됨
- 채팅에 메시지가 전달되지 않음

### 원인: 누락되거나 잘못 구성된 Delivery

격리된 세션 작업은 결과를 알리기 위해 명시적인 delivery 구성이 필요합니다.

### 수정 방법

`delivery` 구성 추가:

```json
{
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Summarize today's news"
  },
  "delivery": {
    "mode": "announce",
    "channel": "telegram"  // or your channel
  }
}
```

## 일반적인 문제 #5: 시간대 혼동

### 증상
- 작업이 잘못된 시간에 실행됨
- 오전 8:00 작업이 오전 3:00 또는 자정에 실행됨
- 스케줄은 올바르게 보이지만 타이밍이 맞지 않음

### 원인: UTC vs 현지 시간

OpenClaw는 기본적으로 UTC를 사용합니다. 다음과 같이 작성하면:
```json
{ "kind": "at", "at": "2026-02-10T08:00:00" }
```
이것은 **UTC** 오전 8:00이며, 현지 시간이 아닙니다.

### 수정 방법

**옵션 1:** 스케줄에 시간대 추가:
```json
{
  "kind": "cron",
  "expr": "0 8 * * *",
  "tz": "Asia/Shanghai"  // or America/New_York, Europe/London, etc.
}
```

**옵션 2:** 시간대 오프셋이 있는 ISO 형식 사용:
```json
{
  "kind": "at",
  "at": "2026-02-10T08:00:00+08:00"
}
```

## 디버깅 체크리스트

위의 방법으로 문제가 해결되지 않으면 이 체크리스트를 따라 진행하세요:

### 1. Gateway가 실행 중인지 확인
```bash
openclaw gateway status
# Should show "running"
```

### 2. 작업 구성 확인
```bash
openclaw cron list --include-disabled
# Verify schedule, payload, and enabled status
```

### 3. 최근 실행 검토
```bash
openclaw cron runs --id <job-id> --limit 10
# Check status of recent executions
```

### 4. 오류 로그 확인
```bash
tail -f ~/.openclaw/logs/gateway.log | grep -E "cron|scheduler|heartbeat"
```

### 5. 수동 트리거 테스트
```bash
openclaw cron run --id <job-id>
# Forces immediate execution to test if job works
```

### 6. JSON 구문 검증
```bash
# Check your config file for JSON errors
cat ~/.openclaw/openclaw.json | jq .
```

## 예방 팁

1. **스케줄에 의존하기 전에 항상 `openclaw cron run`으로 테스트**
2. **스케줄러 이벤트를 캡처하기 위한 로깅 설정**
3. **모든 스케줄에 명시적 시간대 사용**
4. **heartbeat로 모니터링** - 조용한 실패를 포착하기 위해 매일 "I'm alive" 작업 설정
5. **OpenClaw를 최신 상태로 유지** - 많은 스케줄러 버그가 새 버전에서 수정됨
