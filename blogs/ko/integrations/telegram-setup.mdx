# OpenClaw용 Telegram 봇 설정: 완벽 가이드

Telegram은 봇 통합을 위한 가장 인기 있는 메시징 플랫폼 중 하나로, 강력한 API와 뛰어난 개발자 경험을 제공합니다. OpenClaw 인스턴스를 Telegram과 통합하면 안전한 암호화 메시지를 통해 Claude와 상호작용하고, 기기 간 대화를 관리하며, 인라인 키보드, 파일 공유, 그룹 채팅 지원과 같은 Telegram의 풍부한 기능을 활용할 수 있습니다.

이 포괄적인 가이드는 BotFather를 사용하여 봇을 만드는 것부터 실제 GitHub 보고서를 기반으로 한 일반적인 문제 해결까지, 완전한 **openclaw telegram bot setup(OpenClaw Telegram 봇 설정)** 프로세스를 안내합니다. 빠른 로컬 개발을 위해 폴링 모드를 설정하든, 프로덕션 배포를 위해 webhook을 구성하든, 이 가이드가 도와드립니다.

## 사전 요구 사항

**openclaw telegram bot setup**을 시작하기 전에 다음을 확인하세요:

- **OpenClaw 설치 및 실행 중** - 버전 0.5.0 이상 권장
- **Telegram 계정** - 모바일, 데스크톱 또는 웹에서 사용 가능
- **기본적인 터미널/명령줄 지식** - OpenClaw 명령 실행을 위해
- **선택 사항: 공개 HTTPS 엔드포인트** - webhook 모드에만 필요

## 단계 1: BotFather로 봇 생성

BotFather는 다른 봇을 생성하고 관리하기 위한 Telegram의 공식 봇입니다. 이것이 모든 **openclaw telegram bot setup**의 시작점입니다.

### 봇 생성

1. Telegram을 열고 **@BotFather** 검색
2. 대화를 시작하고 `/newbot` 전송
3. 안내에 따르세요:
   - **봇 이름**: 표시 이름 선택 (예: "My OpenClaw Assistant")
   - **봇 사용자명**: "bot"으로 끝나야 합니다 (예: "my_openclaw_bot")
4. BotFather가 **봇 토큰**을 반환합니다 - `123456789:ABCdefGHIjklMNOpqrsTUVwxyz`와 같은 문자열

**⚠️ 보안 경고**: 봇 토큰은 비밀번호와 같습니다. 버전 관리에 커밋하거나 공개적으로 공유하지 마세요. 이 토큰을 가진 사람은 누구나 봇을 제어할 수 있습니다.

### 필수 봇 설정

BotFather를 통해 이러한 중요한 설정을 구성하세요:

**프라이버시 모드** (`/setprivacy`):
- 봇이 그룹 채팅의 모든 메시지를 볼 수 있도록 하려면 프라이버시 모드 **비활성화**
- 봇이 언급되거나 직접 답장된 메시지만 보도록 하려면 **활성화** (기본값)

**그룹 권한** (`/setjoingroups`):
- 봇을 그룹 대화에 추가하려면 **활성화**
- 비공개 일대일 상호작용만 하려면 **비활성화**

**명령** (`/setcommands`):
선택적으로 명령 제안을 설정할 수 있습니다:
```
status - OpenClaw 상태 확인
help - 사용 가능한 명령 표시
reset - 대화 컨텍스트 재설정
```

## 단계 2: OpenClaw 구성

봇 토큰을 얻었으니 이제 **openclaw telegram** 통합을 구성할 차례입니다.

### 기본 구성

**telegram bot polling mode(Telegram 봇 폴링 모드)**를 시작하기 위한 최소 구성:

1. OpenClaw 게이트웨이가 실행 중이면 중지하세요:
```bash
openclaw gateway stop
```

2. OpenClaw 구성 파일 편집 (일반적으로 `~/.openclaw/config.yaml` 또는 유사):

```yaml
plugins:
  entries:
    telegram:
      enabled: true
      token: "YOUR_BOT_TOKEN_HERE"
      dmPolicy: "pair-once"
```

**구성 옵션 설명:**

- `enabled`: Telegram 플러그인을 활성화하려면 `true`여야 합니다
- `token`: BotFather의 봇 토큰
- `dmPolicy`: 다이렉트 메시지 동작 제어
  - `"pair-once"`: 사용자는 페어링을 시작해야 하며, 그 후 자유롭게 채팅할 수 있습니다 (권장)
  - `"open"`: 누구나 페어링 없이 봇에 메시지를 보낼 수 있습니다
  - `"allowlist-only"`: 사전 승인된 사용자만 상호작용할 수 있습니다

### 고급 옵션

프로덕션 배포 또는 다중 사용자 시나리오의 경우 이러한 추가 설정을 고려하세요:

```yaml
plugins:
  entries:
    telegram:
      enabled: true
      token: "YOUR_BOT_TOKEN_HERE"
      dmPolicy: "allowlist-only"
      allowlist:
        - 123456789  # 귀하의 Telegram 사용자 ID
        - 987654321  # 다른 승인된 사용자 ID
      groupPolicy: "allowlist-only"
      groupAllowlist:
        - -1001234567890  # 그룹 채팅 ID (음수 접두사 주의)
      streamMode: true
```

**고급 옵션 설명:**

- `allowlist`: 봇과의 상호작용이 허용된 Telegram 사용자 ID 배열
- `groupPolicy`: `dmPolicy`와 동일한 옵션이지만 그룹 채팅용
- `groupAllowlist`: 봇이 응답할 수 있는 그룹 채팅 ID 배열
- `streamMode`: `true`일 때 긴 응답에 대해 더 나은 UX를 위해 실시간 청크로 응답 전송

**Telegram 사용자 ID 찾기:**

사용자 ID를 찾는 가장 쉬운 방법은 봇에 메시지를 보내고 OpenClaw 로그를 확인하거나 Telegram의 @userinfobot과 같은 봇을 사용하는 것입니다.

## 단계 3: 연결 모드 선택

OpenClaw는 Telegram 메시지를 수신하는 두 가지 방법을 지원합니다: **폴링**과 **webhook**. 각각 고유한 장점과 단점이 있습니다.

### 옵션 A: 폴링 모드 (기본값)

**작동 방식:**
폴링 모드는 OpenClaw가 주기적으로 Telegram 서버에 새 메시지가 있는지 확인하는 방식으로 작동하며, 일반적으로 1-3초마다 확인합니다. 이것이 기본 모드이며 추가적인 **openclaw telegram webhook config(OpenClaw Telegram webhook 구성)**가 필요하지 않습니다.

**장점:**
- ✅ **간단한 설정** - 기본 구성으로 바로 작동
- ✅ **공개 IP 불필요** - 개발 및 로컬 테스트에 완벽
- ✅ **방화벽 내에서 작동** - 인바운드 네트워크 구성 불필요
- ✅ **디버그하기 쉬움** - 모든 로직이 로컬 프로세스에서 실행

**단점:**
- ⚠️ **약간의 지연** - 메시지 수신 전 1-3초 지연
- ⚠️ **리소스 사용** - 지속적인 폴링이 CPU 및 네트워크 대역폭 소비
- ⚠️ **확장 제한** - 높은 트래픽 시나리오에는 이상적이지 않음

**구성:**
폴링 모드는 기본적으로 활성화되어 있습니다. 단계 2의 기본 설정 외에 추가 구성이 필요하지 않습니다.

### 옵션 B: Webhook 모드

**작동 방식:**
Webhook 모드는 Telegram이 HTTPS POST 요청을 통해 새 메시지를 OpenClaw 인스턴스로 직접 푸시하도록 구성합니다. 공개적으로 액세스 가능한 엔드포인트가 필요합니다.

**장점:**
- ✅ **즉시 전달** - 폴링 지연 없음, 메시지가 즉시 도착
- ✅ **리소스 효율적** - 지속적인 폴링 오버헤드 없음
- ✅ **확장 가능** - 프로덕션 및 높은 트래픽 시나리오에 적합

**단점:**
- ⚠️ **공개 HTTPS 엔드포인트 필요** - Telegram 서버에서 액세스 가능해야 함
- ⚠️ **SSL 인증서 필요** - 자체 서명 인증서는 지원되지 않음
- ⚠️ **더 복잡한 설정** - 리버스 프록시 또는 클라우드 호스팅 필요

**구성:**

```yaml
plugins:
  entries:
    telegram:
      enabled: true
      token: "YOUR_BOT_TOKEN_HERE"
      dmPolicy: "pair-once"
      webhook:
        enabled: true
        url: "https://your-domain.com/telegram/webhook"
        secretToken: "your-random-secret-string-here"
        port: 8443  # 선택사항, 기본값은 OpenClaw의 메인 포트
```

**Webhook 설정 단계:**

1. **비밀 토큰 생성** webhook 검증용:
```bash
openssl rand -hex 32
```

2. **리버스 프록시 구성** (nginx 예시):
```nginx
location /telegram/webhook {
    proxy_pass http://localhost:3000/telegram/webhook;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}
```

3. **SSL이 구성되었는지 확인** - Telegram은 유효한 인증서가 있는 HTTPS를 요구합니다 (Let's Encrypt가 잘 작동함)

4. **OpenClaw 재시작** - webhook이 자동으로 Telegram에 등록됩니다

### 폴링 vs Webhook 비교

| 기능 | 폴링 모드 | Webhook 모드 |
|------|----------|-------------|
| **설정 복잡도** | 간단 ✅ | 중간 ⚠️ |
| **공개 IP 필요** | 아니오 ✅ | 예 ⚠️ |
| **메시지 지연** | 1-3초 ⚠️ | 즉시 ✅ |
| **리소스 사용** | 높음 ⚠️ | 낮음 ✅ |
| **최적 용도** | 개발, 로컬 테스트 | 프로덕션, 확장 |
| **디버깅** | 더 쉬움 ✅ | 중간 ⚠️ |

**권장 사항:** 초기 설정 및 테스트에는 폴링 모드로 시작하세요. 프로덕션에 배포하거나 메시지 지연이 문제가 될 때 webhook으로 마이그레이션하세요.

## 단계 4: 봇 테스트

구성이 완료되면 OpenClaw 게이트웨이를 시작하고 통합을 테스트하세요:

### OpenClaw 시작

```bash
openclaw gateway start
```

시작 로그에서 Telegram 초기화를 확인하세요:
```
[INFO] Telegram plugin enabled
[INFO] Telegram: Polling mode active
[INFO] Bot username: @my_openclaw_bot
```

### 첫 번째 메시지 보내기

1. Telegram을 열고 봇을 찾으세요 (생성한 사용자명 검색)
2. `/start`를 보내 대화 시작
3. `dmPolicy: "pair-once"`를 구성한 경우 페어링 프로세스를 완료해야 합니다
4. "안녕하세요!"와 같은 테스트 메시지 전송
5. OpenClaw 인스턴스가 Claude를 통해 응답해야 합니다

### 상태 확인

OpenClaw 상태를 확인하여 **telegram bot polling mode**가 작동하는지 확인하세요:

```bash
openclaw status
```

Telegram이 연결되었음을 나타내는 출력이 표시되어야 합니다:
```
Plugins:
  telegram: ✓ Connected (@my_openclaw_bot)
```

### 페어링 플로우 이해

`dmPolicy: "pair-once"`를 구성한 경우:

1. 사용자가 봇에 첫 번째 메시지를 전송
2. OpenClaw가 페어링 요청 생성
3. 사용자는 페어링을 확인해야 합니다 (정확한 플로우는 OpenClaw 버전에 따라 다름)
4. 페어링 후 해당 사용자의 모든 메시지가 수락됩니다

이 보안 모델은 Claude 인스턴스에 대한 무단 액세스를 방지합니다.

## 일반적인 문제 해결

OpenClaw 커뮤니티가 보고한 실제 GitHub issue를 기반으로 가장 일반적인 문제와 해결 방법을 소개합니다.

### 문제: 봇이 메시지를 수신하지 않음 (폴링 모드)

**GitHub Issue [#6665](https://github.com/openclaw/openclaw/issues/6665) 기반**

**증상:**
- `openclaw status`가 Telegram 플러그인을 "✓ Connected"로 표시
- Telegram에서 봇이 온라인 상태인 것을 확인할 수 있음
- 봇에 보낸 메시지가 응답을 받지 못함
- 검색 기록에 **openclaw telegram not receiving messages** 나타남

**진단 단계:**

1. **봇 토큰이 올바른지 확인:**
```bash
grep -A 5 "telegram:" ~/.openclaw/config.yaml
```

2. **webhook이 폴링을 방해하는지 확인:**
Telegram은 두 모드를 동시에 허용하지 않습니다. 이전에 webhook이 구성된 경우 폴링이 차단됩니다.

webhook 상태 테스트:
```bash
curl "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/getWebhookInfo"
```

값이 있는 `url` 필드가 표시되면 webhook 모드가 활성화되어 폴링을 차단하고 있습니다.

3. **사용자 ID 및 allowlist 구성 확인:**
`dmPolicy: "allowlist-only"`를 사용하는 경우 Telegram 사용자 ID가 allowlist에 있는지 확인하세요.

봇에 메시지를 보내고 OpenClaw 로그를 확인하여 사용자 ID 찾기:
```bash
openclaw gateway logs | grep "Telegram user"
```

4. **플러그인이 실제로 활성화되어 있는지 확인:**
```bash
openclaw config get plugins.entries.telegram.enabled
```

`true`를 반환해야 합니다.

**해결 방법:**

**해결 방법 A: webhook을 제거하여 폴링 활성화**
```bash
curl "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/deleteWebhook?drop_pending_updates=true"
```

**해결 방법 B: 게이트웨이 완전히 재시작**
```bash
openclaw gateway stop
sleep 2
openclaw gateway start
```

**해결 방법 C: allowlist 구성 확인 및 수정**
```yaml
plugins:
  entries:
    telegram:
      enabled: true
      token: "YOUR_BOT_TOKEN"
      dmPolicy: "allowlist-only"
      allowlist:
        - 123456789  # 실제 사용자 ID를 여기에 추가
```

**해결 방법 D: 대기 중인 업데이트 제거**

오래된 메시지가 큐를 막을 수 있습니다:
```bash
curl "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/getUpdates?offset=-1"
```

해결 방법을 구현한 후 봇에 새 메시지를 보내 테스트하세요.

### 문제: 네트워크 중단 중 게이트웨이 충돌

**GitHub Issue [#6676](https://github.com/openclaw/openclaw/issues/6676) 기반**

**증상:**
- OpenClaw 게이트웨이가 정상적으로 실행 중
- 짧은 네트워크 중단 (WiFi 연결 끊김, ISP 문제, VPN 재연결)
- 게이트웨이가 충돌하며 오류 발생: `Unhandled promise rejection: fetch failed`
- 네트워크 복구 후 **telegram bot not working**

**근본 원인:**

이전 버전의 OpenClaw Telegram 플러그인은 폴링 요청 중 네트워크 오류를 제대로 처리하지 못해 처리되지 않은 promise 거부로 Node.js 프로세스가 충돌했습니다.

**해결 방법:**

**해결 방법 A: 최신 OpenClaw 버전으로 업그레이드 (권장)**

이 문제는 OpenClaw v0.5.2 이상에서 수정되었습니다:
```bash
npm install -g openclaw@latest
# 또는
brew upgrade openclaw  # Homebrew를 통해 설치한 경우
```

버전 확인:
```bash
openclaw --version
```

**해결 방법 B: 자동 재시작 구성 (systemd)**

Linux 서버의 경우 충돌 시 자동으로 재시작하도록 systemd를 구성하세요:

`/etc/systemd/system/openclaw.service` 생성:
```ini
[Unit]
Description=OpenClaw Gateway
After=network.target

[Service]
Type=simple
User=youruser
WorkingDirectory=/home/youruser
ExecStart=/usr/local/bin/openclaw gateway start --no-daemon
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

활성화 및 시작:
```bash
sudo systemctl enable openclaw
sudo systemctl start openclaw
```

**해결 방법 C: 재시작 루프 스크립트 사용 (임시 해결책)**

systemd가 없는 개발 또는 시스템의 경우:

```bash
#!/bin/bash
while true; do
  openclaw gateway start
  echo "Gateway crashed, restarting in 5 seconds..."
  sleep 5
done
```

`run-openclaw.sh`로 저장하고 실행 가능하게 만들어 실행:
```bash
chmod +x run-openclaw.sh
./run-openclaw.sh
```

**예방:**

- OpenClaw를 최신 버전으로 유지
- 조기 경고 신호를 위해 게이트웨이 로그 모니터링
- 프로덕션 환경에서 프로세스 매니저 (systemd, PM2, supervisord) 사용
- 사용자 정의 플러그인을 사용하는 경우 네트워크 재시도 로직 구성

### 문제: Markdown 표로 인한 빈 메시지

**GitHub Issue [#6652](https://github.com/openclaw/openclaw/issues/6652) 기반**

**증상:**
- Claude가 markdown 표를 포함한 응답 생성
- Telegram API가 오류 반환: `message text is empty`
- 사용자가 응답을 받지 못하거나 오류 메시지를 받음
- 표를 포함하는 응답에서만 문제 발생

**근본 원인:**

Telegram의 markdown 파서는 복잡한 형식에 제한이 있습니다. OpenClaw가 markdown 표 (`|` 파이프 문자 및 정렬 마커 사용)를 보내려고 하면 Telegram의 파서가 전체 메시지를 무효로 제거하여 빈 메시지 오류가 발생할 수 있습니다.

**해결 방법:**

**해결 방법 A: Telegram 응답에서 표 피하기**

Telegram의 표 형식을 피하도록 OpenClaw 프롬프트를 구성하세요:

```yaml
plugins:
  entries:
    telegram:
      enabled: true
      systemPrompt: |
        Telegram 응답 형식을 지정할 때:
        - 표 대신 글머리 기호 목록 사용
        - 간단한 markdown 사용 (굵게, 기울임꼴, 코드)
        - 복잡한 형식 구조 피하기
```

**해결 방법 B: 글머리 기호 목록 대신 사용**

표 데이터를 구조화된 목록으로 변환:

❌ **사용하지 마세요:**
```markdown
| 기능 | 상태 |
|------|------|
| 폴링 | 활성 |
| Webhook | 비활성 |
```

✅ **대신 사용:**
```markdown
**연결 상태:**
• 폴링: 활성 ✅
• Webhook: 비활성 ⚠️
```

**해결 방법 C: 외부 형식 콘텐츠에 링크**

복잡한 데이터의 경우 대신 링크를 보내세요:
```markdown
자세한 비교 표를 준비했습니다:
👉 https://your-domain.com/comparison

요약:
• 기능 A: 사용 가능
• 기능 B: 곧 출시
```

**해결 방법 D: HTML 모드 사용 (고급)**

OpenClaw를 Markdown 대신 HTML 모드로 Telegram 메시지를 보내도록 구성할 수 있습니다:

```yaml
plugins:
  entries:
    telegram:
      enabled: true
      parseMode: "HTML"  # "Markdown" 대신
```

그런 다음 HTML 표 사용:
```html
<table>
  <tr><th>기능</th><th>상태</th></tr>
  <tr><td>폴링</td><td>활성</td></tr>
</table>
```

**참고:** HTML 모드는 복잡한 형식 지원이 더 좋지만 특수 문자의 HTML 안전 이스케이프가 필요합니다.

### 문제: 터미널에서 QR 코드 렌더링 안 됨

**GitHub Issue [#6640](https://github.com/openclaw/openclaw/issues/6640) 기반**

**증상:**
- 초기 설정 중 OpenClaw가 페어링용 QR 코드를 표시
- QR 코드가 깨진 문자 또는 상자로 나타남
- macOS Terminal.app 및 일부 Linux 터미널에서 특히 일반적
- 초기 설정이 어려워짐

**근본 원인:**

QR 코드는 Unicode 블록 문자를 사용하여 렌더링됩니다. 일부 터미널 에뮬레이터는 필요한 Unicode 범위를 제대로 지원하지 않거나 글꼴 렌더링 문제가 있습니다.

**해결 방법:**

**해결 방법 A: 현대적인 터미널 에뮬레이터 사용**

뛰어난 Unicode 지원을 제공하는 권장 터미널:

- **macOS**: [iTerm2](https://iterm2.com/) 또는 [Kitty](https://sw.kovidgoyal.net/kitty/)
- **Linux**: [Kitty](https://sw.kovidgoyal.net/kitty/), [Alacritty](https://alacritty.org/) 또는 GNOME Terminal
- **Windows**: [Windows Terminal](https://aka.ms/terminal) 또는 [WezTerm](https://wezfurlong.org/wezterm/)

**해결 방법 B: 적절한 Unicode 지원 글꼴 구성**

전체 Unicode 블록 문자 지원이 있는 글꼴을 설치하고 구성하세요:

1. [JetBrains Mono](https://www.jetbrains.com/lp/mono/) 또는 [Fira Code](https://github.com/tonsky/FiraCode) 설치
2. 환경 설정에서 터미널 글꼴로 설정
3. 터미널 재시작 후 다시 시도

**해결 방법 C: QR 코드 없이 수동 구성**

QR 코드 페어링을 완전히 건너뛰고 수동으로 구성:

```bash
# QR 코드를 스캔하는 대신 직접 구성 편집
openclaw config set plugins.entries.telegram.token "YOUR_BOT_TOKEN"
openclaw config set plugins.entries.telegram.enabled true
```

**해결 방법 D: 웹 기반 설정 사용 (사용 가능한 경우)**

일부 OpenClaw 버전은 웹 기반 구성 UI를 제공합니다:

```bash
openclaw gateway config --web
```

터미널 QR 코드 없이 Telegram을 구성할 수 있는 브라우저 인터페이스가 열립니다.

## 보안 모범 사례

**openclaw telegram bot setup**을 보호하는 것은 귀하를 대신하여 Claude AI에 대한 액세스를 제공하기 때문에 매우 중요합니다.

### 1. 봇 토큰 보호

- **토큰을 git에 커밋하지 마세요** - 버전 관리 외부의 환경 변수 또는 구성 파일 사용
- **토큰을 주기적으로 교체** - 손상이 의심되면 새 봇 생성
- **비밀 관리 사용** - HashiCorp Vault, AWS Secrets Manager 또는 암호화된 구성 파일과 같은 도구

### 2. 액세스 제어에 Allowlist 사용

프로덕션에서는 항상 allowlist를 구성하세요:

```yaml
plugins:
  entries:
    telegram:
      dmPolicy: "allowlist-only"
      allowlist:
        - 123456789  # 승인된 사용자만
```

### 3. 봇 권한 정기적으로 검토

BotFather를 통해 봇이 할 수 있는 작업 확인:
- `/mybots` → 봇 선택 → 설정 검토
- 사용하지 않는 기능 비활성화 (그룹, 인라인 모드 등)
- 필요하지 않은 경우 그룹 권한 제한

### 4. Webhook 비밀 토큰 사용

**openclaw telegram webhook config**를 사용할 때 항상 비밀 토큰을 설정하세요:

```yaml
webhook:
  secretToken: "use-a-long-random-string-here"
```

Telegram은 이를 `X-Telegram-Bot-Api-Secret-Token` 헤더에 포함하여 요청이 진짜인지 확인할 수 있습니다.

### 5. 사용량 및 로그 모니터링

정기적으로 봇을 사용하는 사람을 검토하세요:
```bash
openclaw gateway logs | grep "Telegram message from"
```

비정상적인 활동 패턴에 대한 알림을 설정하세요.

### 6. OpenClaw를 최신 상태로 유지

보안 수정은 정기적으로 릴리스됩니다:
```bash
openclaw update
# 또는
npm update -g openclaw
```
