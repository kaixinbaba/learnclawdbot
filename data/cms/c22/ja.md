# OpenClaw ロードマップ 2026：オープンソース AI ゲートウェイの今後

OpenClaw の進化は目覚ましいものがあります。わずか約 2 年で、シンプルな Claude の CLI ラッパーから、複数のモデル、MCP サーバー、音声入力、ブラウザ自動化、プラグインエコシステムをサポートする本格的なセルフホスト AI ゲートウェイへと成長しました。変化のペースはあまりに速く、半年前には高度な機能と思っていたものが、今ではデフォルト設定になっています。

この記事では、2026 年の展望を紹介します。ロードマップで確認済みの機能、活発に開発が進むコミュニティ主導のプロジェクト、そしてエコシステム全体の動作方法に影響を与えるプロトコルレベルの変更についてです。情報源は、公開されている GitHub ロードマップ、コミュニティの Discord での議論、そしてコントリビューターとの直接の会話です。

## プロトコル層：MCP 2.0

モデルコンテキストプロトコル（Model Context Protocol）は OpenClaw エコシステムで最も重要なプロトコルであり、その仕様は 2026 年に大きく進化します。

**MCP 2.0 の主な変更点：**

- **ストリーミングツールレスポンス**：現在、ツール呼び出しは同期的で、モデルは完全な結果を受け取るまで待機します。MCP 2.0 ではストリーミングが追加され、長時間実行されるツールが増分更新を送信できるようになります。ファイルの監視、ログのモニタリング、時間のかかる操作を実行するツールにとって重要な改善です。

- **バージョン管理付きツールスキーマ**：ツールスキーマがバージョン互換性を宣言できるようになり、サーバーが API を更新した際のグレースフルデグラデーションが実現します。

- **マルチモーダルツール入力**：MCP 2.0 では、テキストに加えて画像と音声入力を受け付けるツールをサポートします。これにより、プロトコル内でネイティブに動作するビジョンベースの自動化ツール（スクリーンショット分析、OCR）が利用可能になります。

- **双方向サーバープッシュ**：サーバーがモデルクライアントに自発的なイベントを送信できるようになります。リアルタイム通知が実現し、受信トレイを監視する MCP サーバーが緊急メールの到着時に通知をプッシュできます。モデルがポーリングする必要はありません。

OpenClaw は 2026 年 Q2 に MCP 2.0 対応を約束しています。既存の MCP サーバーの移行は、ほとんどのサーバーで破壊的変更なしに行えるはずです。2.0 仕様は 1.x ツールとの後方互換性を維持しています。

## モデル統合の更新

**シンキングモデルのサポート**：OpenAI の o3/o4 シリーズと Anthropic の拡張シンキング機能を持つ Claude モデルは、モデルの推論プロセスへのアクセスを提供します。OpenClaw はこれらの推論トレース専用の表示を追加します。最終的な回答とは分離された折りたたみ可能なセクションとして UI に表示されます。

**コンピューター使用の自動化**：Anthropic のコンピューター使用 API（スクリーンショットと入力イベントを通じてデスクトップを制御する機能）が OpenClaw の主要機能として統合されます。現在の Playwright 統合はブラウザの自動化をカバーしており、今後のコンピューター使用機能はこれをフルデスクトップ制御に拡張します。2026 年 Q3 を予定しています。

**ローカルモデルサポートの改善**：Ollama 統合が自動モデル推薦機能で拡張されます。タスクを説明すると、OpenClaw がそのタスクタイプに最も適した利用可能なローカルモデルを提案します。単に利用可能なモデルではなく、最適なモデルを選択します。

**マルチプロバイダールーティング**：2026 年 Q2 に計画されている重要なアーキテクチャの追加機能は、インテリジェントなリクエストルーティングです。プロバイダーのプールとルーティングポリシー（コスト、レイテンシ、品質）を定義すると、OpenClaw が各リクエストに対して最適な利用可能プロバイダーを自動的に選択します。オフライン時はクラウドからローカルにフォールバックし、シンプルなタスクは安価なモデルが、複雑なタスクは高性能なモデルが処理します。

```yaml
# Planned config syntax for multi-provider routing
routing:
  policy: quality-cost-balanced
  providers:
    - name: claude-sonnet
      priority: 1
      conditions:
        - task_type: complex_reasoning
    - name: mistral-local
      priority: 2
      conditions:
        - network: offline
        - task_type: simple_qa
    - name: deepseek-v3
      priority: 3
      conditions:
        - cost_target: low
```

## プラットフォーム機能

**OpenClaw Agents（永続的なバックグラウンドエージェント）**：現在のインタラクションモデルはリクエスト・レスポンス型です。あなたが質問し、モデルが回答します。計画中の Agents 機能では、永続的なバックグラウンドプロセスが追加されます。AI エージェントが継続的に実行され、条件を監視し、アクションを実行します。例えば、CI/CD パイプラインを監視して失敗をまとめるモニタリングエージェントや、定期的にナレッジベースを更新するリサーチエージェントが考えられます。

これはアーキテクチャ的に重要な変化です。OpenClaw がセッションをまたいでエージェントの状態を管理し、エージェント間の通信を処理し、実行中のエージェントの監視インターフェースを提供することが求められます。目標：2026 年 Q4。

**プラグインマーケットプレイスの改善**：ClawdHub（OpenClaw プラグインディレクトリ）が大幅に刷新されます。主な追加機能：
- 掲載前の自動プラグインセキュリティスキャン
- プラグインの評価と使用統計
- `openclaw plugin install <plugin-name>` によるワンクリックインストール
- プラグイン互換性マトリックス（どのモデルがどのプラグインをサポートするか）

**チームワークスペース**：現在、マルチユーザーの OpenClaw デプロイメントでは、共有コンテキストと権限を手動で設定する必要があります。計画中のチームワークスペース機能では、組織レベルの設定が導入されます。共有システムプロンプト、集中管理された API キー管理、そして適切な管理インターフェースを通じたユーザーごとの権限スコープ設定が実現します。

**エクスポートとインポート**：セッション履歴を標準フォーマット（Markdown、JSON）にエクスポートし、アーカイブ、共有、OpenClaw インスタンス間の移行に利用できます。AI アシスト意思決定の記録を保持したい組織にとって便利な機能です。

## パフォーマンスの改善

**コンテキストストリーミングの最適化**：現在のセッションでは、各リクエストで会話履歴全体が読み込まれます。2026 年 Q2 には増分コンテキスト読み込みが導入され、新しいメッセージのみが転送され、完全な履歴はサーバーサイドのキャッシュに保持されます。長いセッションでは、レイテンシを 40〜60% 削減できます。

**MCP サーバー接続プーリング**：現在、各 MCP サーバーはリクエストごとに新しい接続を取得します。2026 年 Q1 に計画されている接続プーリングにより、頻繁に使用されるサーバーへの永続的な接続が維持されます。ツールを多用するワークフローのレイテンシオーバーヘッドが削減されます。

**オフラインファーストアーキテクチャ**：OpenClaw の UI 機能の多くが、インターネット接続なしで動作するようになります。現在、完全にローカルなデプロイメントでも、一部の UI 要素は OpenClaw のアップデートサーバーへのアクセスを必要とします。オフラインファーストのリファクタリングにより、ローカルデプロイメントが真に独立して動作することが保証されます。

## 注目のコミュニティプロジェクト

**openclaw-langchain**：LangChain のエージェントエコシステムを OpenClaw の MCP インターフェース経由で公開するブリッジです。まだ実験的ですが、追加設定なしに OpenClaw CLI から LangChain ツールを使用できます。

**HA-OpenClaw**：Home Assistant のアドオンとして、OpenClaw HA MCP サーバーと HA のアドオンストアにある設定 UI をパッケージ化したものです。HA ユーザーのホームオートメーション統合をクリック操作で実現します。

**openclaw-raycast**：macOS 上の Raycast ランチャーを通じて OpenClaw を公開する Raycast 拡張機能です。キーボードショートカットでシステムのどこからでも OpenClaw を呼び出せます。

**openclaw-vscode**：開発中の VS Code 拡張機能です。VS Code 内に直接 OpenClaw パネルを提供し、エディターを離れることなく AI アシスタントに質問できます。GitHub Copilot とは異なり、ローカルモデルを含む OpenClaw がサポートするあらゆるモデルで動作します。

**OpenClaw iOS**：セルフホストの OpenClaw インスタンスに接続する初期段階のモバイルクライアントです。初期バージョンはシンクライアントで、本質的には OpenClaw API のモバイル UI です。音声入力がモバイルユースケースの主要な操作モードです。

## 最も注目している機能

マルチプロバイダールーティング機能と永続的なエージェントが、最も実際的な影響をもたらすと考えています。ルーティングはコスト構造の異なる複数のモデルサブスクリプションを管理するという現実的な問題を解決します。現在は手動でこれをこなしています。タスクごとにどのモデルを使うかを選択するのに、自動化によって相当な時間が節約できるはずです。

永続的なエージェントはより大きなアーキテクチャ的な転換です。今は、すべての OpenClaw とのやり取りに私が存在し、積極的にクエリを実行する必要があります。バックグラウンドエージェントは、私の直接的な関与なしに監視、要約、アラートを実行できます。これにより OpenClaw は、私が使うツールから、私のために働くインフラへと変わります。

MCP 2.0 のストリーミング更新は、エンドユーザーよりも MCP サーバーを構築する開発者にとって重要な意味を持ちますが、間接的な恩恵—より優れたツール統合とより応答性の高い動作—はエコシステム全体で体感できるでしょう。

## コントリビューション

OpenClaw は MIT ライセンスで、積極的にコントリビューションを受け付けています。特に支援が必要な分野：

- **MCP サーバーの実装**：多くの便利なサービスがまだ MCP サーバーを持っていません。日常的に使用しているサービスを OpenClaw から利用したいなら、MCP サーバーを構築することが最も影響力の高いコントリビューションです。
- **ドキュメントとチュートリアル**：プラットフォームが複雑になるにつれ、ドキュメントの品質がより重要になります。
- **テスト**：MCP 統合レイヤーのエンドツーエンドテストは未完成です。この分野へのコントリビューションが信頼性を直接向上させます。
- **モバイルクライアント**：iOS および Android クライアントはコミュニティの取り組みです。モバイル開発者を歓迎します。

ロードマップは GitHub で公開されています。機能リクエストと実装の議論は GitHub Issues で行われています。実装に多大な投資をする前に、コントリビューションのフィードバックを得るための最も迅速な方法は Discord サーバーです。

## よくある質問

**OpenClaw はオープンソースであり続けますか？**

はい。OpenClaw は MIT ライセンスで、コアチームはプロジェクトのオープンソースの性質を維持することを明確に約束しています。ビジネスモデル（存在する場合）は、コアソフトウェアを有料化するのではなく、クラウドホスティングとサポートサービスになると予想されています。

**MCP 2.0 はいつ利用可能になりますか？**

MCP 仕様チームは 2026 年 Q2 に 2.0 仕様を確定させることを目標としています。OpenClaw の実装は仕様に準拠します。OpenClaw の MCP 2.0 サポートは 2026 年 Q3 を予定しており、ベータ版はそれより早く提供されます。

**バックグラウンドエージェント機能には追加のインフラが必要ですか？**

シンプルなエージェント（監視、要約）には、既存の OpenClaw サーバーで十分です。複雑なマルチエージェントワークフローには、専用のコンピューティングリソースが必要になる可能性があります。この機能は、まず既存のハードウェアで小規模に動作するよう設計されています。

**後方互換性はどうなりますか？**

コアチームは後方互換性を真剣に考えています。設定フォーマットや API の破壊的変更には常にバージョン管理が行われ、移行ガイドが提供されます。初期の OpenClaw バージョンから現在に至る移行は一部で困難な部分もありましたが、チームは初期の頃よりも互換性についてより規律を持って取り組んでいます。

2026 年のロードマップは、基盤を固め、今や幅ではなく深さを追求するプラットフォームを反映しています。基本的なコンポーネント——マルチモデルサポート、MCP、プラグイン——は確立されています。次のフェーズでは、これらのコンポーネントをより強力で、より信頼性が高く、YAML 設定ファイルに慣れていないユーザーにも使いやすくすることです。それが正しい方向性です。
