# OpenClaw 2026 路线图：开源 AI 网关的下一步

OpenClaw 的发展速度惊人。在短短约两年时间里，它从一个简单的 Claude CLI 封装工具，演变为一个功能完善的自托管 AI 网关，支持多种模型、MCP 服务器、语音输入、浏览器自动化和插件生态系统。变化的速度之快，让我半年前还觉得是高级功能的东西，现在已经成为默认配置。

本文将展望 2026 年的发展前景：路线图中已确认的功能、正在积极开发中的社区驱动项目，以及将影响整个生态系统运作方式的协议层变化。我的信息来源包括公开的 GitHub 路线图、社区 Discord 讨论，以及与贡献者的直接交流。

## 协议层：MCP 2.0

模型上下文协议（Model Context Protocol）是 OpenClaw 生态系统中最重要的协议，其规范将在 2026 年经历重大演进。

**MCP 2.0 主要变化：**

- **流式工具响应**：目前，工具调用是同步的——模型在收到完整结果之前会一直等待。MCP 2.0 引入了流式传输，使长时间运行的工具可以发送增量更新。这对于监视文件、监控日志或执行耗时操作的工具来说至关重要。

- **带版本控制的工具模式**：工具模式可以声明版本兼容性，当服务器更新其 API 时能够实现优雅降级。

- **多模态工具输入**：MCP 2.0 支持同时接受图像和音频输入与文本的工具，这为在协议内原生运行基于视觉的自动化工具（截图分析、OCR）打开了大门。

- **双向服务器推送**：服务器可以主动向模型客户端发送事件，无需客户端轮询。这实现了实时通知功能——例如，监视收件箱的 MCP 服务器可以在紧急邮件到达时主动推送通知，无需模型反复查询。

OpenClaw 已承诺在 2026 年第二季度支持 MCP 2.0。对于现有 MCP 服务器的迁移，大多数服务器应该不会有破坏性变化——2.0 规范保持了与 1.x 工具的向后兼容性。

## 模型集成更新

**支持思考型模型**：OpenAI 的 o3/o4 系列和 Anthropic 的 Claude 扩展思考模型都提供了访问模型推理过程的能力。OpenClaw 正在为这些推理过程添加专属显示——它们将以可折叠区块的形式呈现在界面中，与最终响应分开显示。

**计算机使用自动化**：Anthropic 的计算机使用 API（通过截图和输入事件控制桌面的能力）正在被整合为 OpenClaw 的一项核心功能。目前的 Playwright 集成覆盖了浏览器自动化；即将推出的计算机使用功能将把这一能力扩展到完整的桌面控制。预计在 2026 年第三季度推出。

**改进的本地模型支持**：Ollama 集成正在扩展，新增自动模型推荐功能。当你描述一项任务时，OpenClaw 将建议最适合该任务类型的本地模型——不只是任意一个可用模型，而是最合适的那个。

**多提供商路由**：计划在 2026 年第二季度推出的一项重要架构新增功能是智能请求路由。你可以定义一组提供商和路由策略（成本、延迟、质量），OpenClaw 将自动为每个请求选择最佳的可用提供商。离线时云端回退到本地；简单任务由廉价模型处理，复杂任务由高性能模型处理。

```yaml
# Planned config syntax for multi-provider routing
routing:
  policy: quality-cost-balanced
  providers:
    - name: claude-sonnet
      priority: 1
      conditions:
        - task_type: complex_reasoning
    - name: mistral-local
      priority: 2
      conditions:
        - network: offline
        - task_type: simple_qa
    - name: deepseek-v3
      priority: 3
      conditions:
        - cost_target: low
```

## 平台功能

**OpenClaw Agents（持久后台代理）**：当前的交互模型是请求-响应模式：你提问，模型回答。计划中的 Agents 功能将新增持久后台进程——AI 代理持续运行、监控条件并采取行动。试想：一个监控你 CI/CD 流水线并汇总失败信息的监控代理，或者一个定期更新知识库的研究代理。

这在架构上意义重大。它要求 OpenClaw 跨会话管理代理状态、处理代理间通信，并为运行中的代理提供监督界面。目标：2026 年第四季度。

**插件市场改进**：ClawdHub（OpenClaw 插件目录）将进行重大改版。主要新增功能：
- 上架前自动进行插件安全扫描
- 插件评分和使用统计
- 通过 `openclaw plugin install <plugin-name>` 一键安装
- 插件兼容性矩阵（哪些模型支持哪些插件）

**团队工作区**：目前，多用户 OpenClaw 部署需要手动配置共享上下文和权限。计划中的团队工作区功能将引入组织级配置——共享系统提示词、集中式 API 密钥管理，以及通过正规管理界面进行的每用户权限范围控制。

**导出与导入**：将会话历史导出为标准格式（Markdown、JSON），用于归档、共享和在 OpenClaw 实例之间迁移。对于希望保留 AI 辅助决策记录的组织来说非常实用。

## 性能改进

**上下文流式传输优化**：目前，每次请求都会加载完整的对话历史。2026 年第二季度将带来增量上下文加载——只传输新消息，完整历史记录保存在服务器端缓存中。对于长会话，这可以将延迟降低 40-60%。

**MCP 服务器连接池**：目前每个 MCP 服务器每次请求都会建立新连接。计划在 2026 年第一季度推出的连接池功能将为频繁使用的服务器维护持久连接，从而降低工具密集型工作流的延迟开销。

**离线优先架构**：OpenClaw 的更多 UI 功能将在没有网络连接的情况下正常工作。目前，即使在完全本地化的部署中，某些 UI 元素仍需访问 OpenClaw 的更新服务器。离线优先重构将确保本地部署真正独立运行。

## 值得关注的社区项目

**openclaw-langchain**：一个通过 OpenClaw 的 MCP 接口暴露 LangChain 代理生态系统的桥接工具。目前仍处于实验阶段，但能让 LangChain 工具从 OpenClaw CLI 使用，无需额外配置。

**HA-OpenClaw**：一个专用的 Home Assistant 插件，将 OpenClaw HA MCP 服务器与 HA 插件商店中的配置界面打包在一起。让家庭自动化集成对 HA 用户来说变成点击操作。

**openclaw-raycast**：一个 Raycast 扩展，通过 macOS 上的 Raycast 启动器暴露 OpenClaw。使用键盘快捷键可从系统任意位置调用 OpenClaw。

**openclaw-vscode**：正在积极开发中的 VS Code 扩展。直接在 VS Code 中提供 OpenClaw 面板，让你无需离开编辑器即可查询 AI 助手。与 GitHub Copilot 不同，它支持任何 OpenClaw 支持的模型，包括本地模型。

**OpenClaw iOS**：早期阶段的移动客户端，可连接到自托管的 OpenClaw 实例。初始版本是一个轻量级客户端——本质上是 OpenClaw API 的移动端界面。语音输入是移动端使用场景的主要交互方式。

## 我最感兴趣的功能

多提供商路由功能和持久代理是我认为将产生最大实际影响的两项变化。路由功能解决了管理多个具有不同成本结构的模型订阅这一现实问题。我目前是手动完成这件事的——为每个任务选择使用哪个模型——自动化将节省大量时间。

持久代理是更大的架构转变。现在，每次 OpenClaw 交互都需要我在场并主动查询。后台代理可以在没有我直接参与的情况下进行监控、汇总和提醒——将 OpenClaw 从我使用的工具转变为为我工作的基础设施。

MCP 2.0 的流式传输更新对构建 MCP 服务器的开发者来说意义更大，而非最终用户，但间接效益——更好的工具集成和更灵敏的行为——将在整个生态系统中都能感受到。

## 参与贡献

OpenClaw 采用 MIT 许可证，积极接受贡献。最需要帮助的领域：

- **MCP 服务器实现**：许多实用服务还没有 MCP 服务器。如果你经常使用某个服务并希望从 OpenClaw 中使用它，构建 MCP 服务器是你能做出的效益最高的贡献。
- **文档和教程**：随着平台复杂性的增加，文档质量变得更加重要。
- **测试**：MCP 集成层的端到端测试尚不完整。在这方面的贡献将直接提高可靠性。
- **移动客户端**：iOS 和 Android 客户端都是社区项目。欢迎移动开发者参与。

路线图已在 GitHub 上公开。功能请求和实现讨论在 GitHub Issues 中进行。在大量投入实现之前，Discord 服务器是获取贡献反馈的最快方式。

## 常见问题

**OpenClaw 会继续保持开源吗？**

是的。OpenClaw 采用 MIT 许可证，核心团队已明确承诺维护项目的开源性质。商业模式（如果有的话）预计将是云托管和支持服务，而非将核心软件设置门槛。

**MCP 2.0 何时可用？**

MCP 规范团队目标在 2026 年第二季度完成 2.0 规范。OpenClaw 的实现遵循该规范。预计 OpenClaw 的 MCP 2.0 支持将在 2026 年第三季度推出，测试版将更早提供。

**后台代理功能是否需要额外的基础设施？**

对于简单代理（监控、汇总），现有的 OpenClaw 服务器已经足够。对于复杂的多代理工作流，你可能需要专用计算资源。该功能被设计为首先在现有硬件的小规模场景下工作。

**向后兼容性如何保证？**

核心团队非常重视向后兼容性。配置格式或 API 的破坏性变更始终有版本控制，并提供迁移指南。从早期 OpenClaw 版本升级到当前版本的过程在某些地方确实有些曲折，但团队在兼容性方面比早期更加严格自律。

2026 年路线图反映了一个已经找到立足点、现在正在深化而非拓宽的平台。基础组件——多模型支持、MCP、插件——已经建立完善。下一阶段是让这些组件更强大、更可靠，并让那些不习惯 YAML 配置文件的用户也能轻松使用。这是正确的方向。
