# Оптимизация управления контекстом OpenClaw для длительных сессий

Контекстное окно — важнейшее ресурсное ограничение при работе с AI-ассистентами. Всё, что модель знает о вашей текущей задаче — история переписки, переданные файлы, предоставленный контекст — должно умещаться в этом окне. Когда оно заполняется, старая информация вытесняется, и модель начинает терять нить разговора.

Большинство людей впервые сталкиваются с этим ограничением при работе с большими кодовыми базами: они передают кучу файлов, просят выполнить сложный рефакторинг, и где-то в середине процесса модель забывает об ограничениях, установленных в самом начале. Качество падает, разговор приходится начинать заново.

Понять, как OpenClaw управляет контекстом, и научиться работать в соответствии с этим управлением, а не вопреки ему — значит превратить раздражающие проблемы с переполнением контекста в решаемые инженерные задачи.

## Как OpenClaw обрабатывает контекст

Контекст OpenClaw состоит из нескольких компонентов, потребляющих tokens:

1. **Системный промпт** — инструкции, определяющие поведение модели (задаются OpenClaw, при необходимости расширяются через конфигурацию)
2. **История разговора** — все сообщения текущей сессии
3. **Загруженные файлы** — любые файлы, переданные через `--files` или включённые иным способом
4. **Результаты вызовов инструментов** — ответы MCP-серверов и встроенных инструментов OpenClaw
5. **Внедрённый контекст** — фоновая информация, явно загруженная с помощью `--context`

Суммарный объём должен укладываться в контекстное окно модели. Для облачных моделей:
- Claude 3.5 Sonnet: 200K tokens
- GPT-4o: 128K tokens
- Gemini 1.5 Pro: 1M tokens (но дорого)

Для локальных моделей:
- Большинство моделей 7B-13B: 8K-32K tokens
- Llama 3.1: до 128K
- Mistral: 32K-64K в зависимости от версии

Приблизительное соотношение: 1 token ≈ 4 символа английского текста. Типичный файл с кодом в 200 строк занимает примерно 2 000-4 000 tokens.

## Просмотр текущего использования контекста

OpenClaw отображает использование tokens несколькими способами:

```bash
# Показать статистику текущей сессии
openclaw status

# Вывод:
# Session: dev-session-001
# Messages: 23
# Tokens used: 47,234 / 200,000 (23.6%)
# Model: claude-3-5-sonnet
# Active files: 4 (12,400 tokens)
```

В ходе разговора строка контекста в интерфейсе OpenClaw отображает использование в реальном времени. Когда она достигает 80%, самое время подумать о стратегиях управления контекстом.

## Стратегия 1: Точечная загрузка файлов

Самая распространённая ошибка — загружать файл целиком, когда нужна лишь его часть. Файл на 2 000 строк обойдётся в 20 000+ tokens. Если вам нужно лишь понять, что он экспортирует, хватит и 200 tokens.

**Загружайте конкретные разделы:**

```bash
# Вместо загрузки всего файла
openclaw chat --files src/api/server.ts "explain the auth middleware"

# Лучше: опишите, что именно вам нужно
openclaw chat "Look at src/api/server.ts and explain only the auth middleware section"
```

Когда вы просите OpenClaw «посмотреть» на файл, а не загружаете его через `--files`, он читает файл и извлекает только нужное. Это зачастую более экономно с точки зрения tokens.

**Используйте glob-паттерны, чтобы загружать только нужные файлы:**

```bash
# Плохо: загружает всё подряд
openclaw chat --files "src/**/*.ts" "fix the authentication bug"

# Лучше: загружаем только файлы, связанные с аутентификацией
openclaw chat --files "src/auth/*.ts,src/middleware/auth*.ts" "fix the authentication bug"
```

## Стратегия 2: Резюмировать перед продолжением

Длинные разговоры быстро накапливают «долг» по tokens. Каждое сообщение в истории чего-то стоит. После продуктивной сессии, в которой была решена задача, резюмируйте результаты, прежде чем переходить к следующему вопросу:

```
Вы: «Давайте подведём итоги того, что мы установили, прежде чем продолжать:
- Баг аутентификации находится в jwt.ts на строке 47
- Исправление предполагает проверку срока действия токена до валидации подписи
- Тестовый файл нужно обновить с учётом нового поведения

Теперь, имея это в виду, займёмся связанной проблемой истечения срока сессии в session.ts»
```

Явное резюме даёт модели компактную точку опоры, заменяя разветвлённый разговор, который привёл к этим выводам. Модель может работать, опираясь на резюме, без необходимости перечитывать все предыдущие сообщения.

## Стратегия 3: Именованные сессии для многодневных проектов

Сессии OpenClaw по умолчанию эфемерны. При закрытии терминала контекст разговора пропадает. Для проектов, растянутых на несколько дней, используйте именованные сессии:

```bash
# Запустить именованную сессию
openclaw chat --session my-refactor-project

# Возобновить её позже
openclaw chat --session my-refactor-project
```

Именованные сессии сохраняют историю разговора на диск. Но сохранение не решает проблему бюджета tokens — у недельной истории переписки будет огромный объём. Решение — использовать сессию для сфокусированной фазы работы, а затем начинать новую сессию для следующей фазы.

**Рабочий процесс с сессиями для многонедельного проекта:**

```
Неделя 1: --session refactor-phase-1
  Цель: разобраться в кодовой базе, выявить проблемные места
  Финал: резюмировать находки, закрыть сессию

Неделя 2: --session refactor-phase-2
  Старт: загрузить резюме 1-й недели как контекст
  Цель: реализовать исправления для выявленных проблем
  Финал: резюмировать внесённые изменения, закрыть сессию

Неделя 3: --session refactor-phase-3
  Старт: загрузить резюме 2-й недели
  Цель: тестирование и граничные случаи
```

## Стратегия 4: Файлы контекста для фоновой информации

Для информации, полезной во множестве разговоров — документация проекта, стандарты кодирования, схемы API — используйте функцию загрузки контекста в OpenClaw:

```bash
# Создать файл контекста со стабильной фоновой информацией
cat > .openclaw-context.md << 'EOF'
# Project Context

This is a Node.js API for a SaaS billing system.
Key constraints:
- Never modify the Stripe webhook handlers without QA sign-off
- All amounts are in cents (no floats for money)
- The legacy V1 API must remain backward compatible
- Use the internal audit logger for all financial operations

Database: PostgreSQL 16
ORM: Drizzle
Auth: JWT with 24h expiry, refresh tokens stored in Redis
EOF

# Загружать в начале каждой сессии
openclaw chat --context .openclaw-context.md "Let's work on the invoice generation feature"
```

Это эффективнее, чем заново объяснять контекст в каждом разговоре. Файл контекста потребляет tokens, но он компактен, и вы полностью контролируете его содержимое.

## Стратегия 5: Паттерн «рабочего пространства»

Для работы с большими кодовыми базами определите рабочее пространство: тщательно подобранный набор наиболее релевантных файлов, комфортно помещающихся в бюджет контекста.

```bash
# Создать конфигурацию рабочего пространства
cat > .openclaw-workspace.yaml << 'EOF'
name: auth-system
files:
  - src/auth/jwt.ts
  - src/auth/session.ts
  - src/middleware/auth.ts
  - src/models/user.ts
  - tests/auth/*.test.ts
context:
  - docs/auth-architecture.md
EOF

# Работать в рамках рабочего пространства
openclaw chat --workspace .openclaw-workspace.yaml "audit the session handling for security issues"
```

Рабочее пространство остаётся согласованным между сессиями. Когда нужно расширить охват, просто обновите файл рабочего пространства.

## Стратегия 6: Итеративное уточнение

Для длинных документов или сложных задач работайте поэтапно, а не пытайтесь сделать всё за один проход:

**Резюме → Структура → Черновик → Доработка**

```
Проход 1: «Прочитай эти 5 исходных файлов и дай мне краткое описание работы каждого в одном абзаце»
Проход 2: «На основе этих описаний набросай план рефакторинга потока аутентификации»
Проход 3: «Теперь реализуем шаг 1 твоего плана — только изменения в валидации JWT»
Проход 4: «Давай проверим, что мы изменили, и протестируем граничные случаи»
```

Каждый проход опирается на компактные результаты предыдущего. К моменту реализации в контексте находятся не полные тексты файлов, а только резюме.

## Мониторинг контекста в автоматизации

При скриптовом использовании API OpenClaw отслеживайте использование контекста программно:

```typescript
const response = await fetch("http://localhost:11434/api/chat", {
  method: "POST",
  body: JSON.stringify({
    model: "claude-3-5-sonnet",
    messages: conversationHistory,
  }),
});

const data = await response.json();

// Проверить использование tokens в ответе
const usage = data.usage;
console.log(`Tokens used: ${usage.prompt_tokens} + ${usage.completion_tokens}`);

// Если приближаемся к лимиту, запустить суммаризацию
if (usage.prompt_tokens > 150000) {
  await summarizeAndCompressHistory(conversationHistory);
}
```

## Работа с переполнением контекста

Когда контекст заполняется несмотря на все усилия:

**Паттерн graceful recovery:**

1. Попросите модель резюмировать текущее состояние: «Прежде чем продолжать, пожалуйста, составь краткое резюме по пунктам: (1) что мы пытались сделать, (2) что мы выяснили, (3) каков следующий шаг»
2. Начните новую сессию
3. Загрузите резюме как начальный контекст

```bash
# Экспортировать резюме из старой сессии
openclaw chat --session old-session "summarize our current state for handoff to a new session" > session-handoff.txt

# Начать новую сессию с эстафетным резюме
openclaw chat --context session-handoff.txt "Continue from the session summary above..."
```

## Особенности контекста для разных моделей

Разные модели по-разному считают tokens:

**Модели Claude** используют токенизатор Anthropic. Код относительно компактен — Claude хорошо обрабатывает синтаксис кода, поэтому файлы с кодом не раздуваются так сильно, как у некоторых других моделей.

**Модели GPT** используют tiktoken. Схожая эффективность для текста на английском, чуть менее эффективны для кода с нестандартным синтаксисом.

**Локальные модели** различаются. Модели на базе Llama используют токенизатор SentencePiece. Азиатские языки могут обходиться дороже в tokens, чем у некоторых облачных моделей со специализированными токенизаторами.

Для многоязычной работы имейте в виду, что китайский и японский текст с некоторыми токенизаторами потребляет в 2-3 раза больше tokens, чем аналогичный текст на английском. По возможности предпочитайте английский для файлов контекста и фоновой информации — даже если конечный результат будет на другом языке.

## Часто задаваемые вопросы

**OpenClaw удалил мою историю разговора — что произошло?**

Безымянные сессии по умолчанию не сохраняются на диск. Используйте `--session session-name` для сохранения разговоров. Сохранённые сессии можно найти в `~/.openclaw/sessions/`.

**Как узнать, что я скоро достигну лимита контекста?**

OpenClaw показывает полосу прогресса в интерфейсе. При использовании API следите за полем `usage.prompt_tokens` в ответах. Установите порог предупреждения (например, 80% от контекстного окна модели) и запускайте суммаризацию до достижения лимита.

**Сброс разговора очищает контекст?**

Да, `openclaw clear` или `/clear` в чате очищают историю разговора. Файлы, загруженные через `--files`, и контекст, загруженный через `--context`, также освобождаются. Следующее сообщение начинается с чистого листа.

**Можно ли увеличить контекстное окно для локальных моделей?**

Размер контекста в Ollama задаётся при загрузке модели. Увеличить его можно, но придётся идти на компромисс между качеством и производительностью. За пределами обученной длины контекста модели (не лимита VRAM) качество деградирует:

```bash
# Установить размер контекста для Ollama
OLLAMA_NUM_CTX=32768 ollama run mistral:7b
```

Превышайте значение по умолчанию, только убедившись через тестирование, что качество остаётся приемлемым.

Управление контекстом — навык, приносящий всё больше пользы по мере освоения. Как только вы интернализируете, как работают tokens и какие стратегии помогают их экономить, работа с большими кодовыми базами становится принципиально эффективнее. Ограничения перестают ощущаться как стены и начинают восприниматься как инженерные задачи, которые можно обойти.
