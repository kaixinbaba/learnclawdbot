# Kev's Dream Team：OpenClawで14+のAIエージェントを并行実行

## 概要

- **カテゴリー：** マルチエージェントオーケストレーション / 生産性
- **対象者：** AIエージェントでワークフローを拡張したい個人開発者、コンテンツクリエイター、小規模チーム
- **ソース：**
  - [OpenClaw ショーケース](https://docs.openclaw.ai/start/showcase)
  - [Orchestrated AI アーティクル](https://github.com/adam91holt/orchestrated-ai-articles)

## 背景

単一エージェントのワークフローは、複雑なマルチステッププロジェクトを処理する際に限界に達します。コンテンツパイプラインの管理、複数のソースにまたがる研究、または公開ワークフローの調整に関係なく、タスクを順番に実行するとボトルネックが発生します。

このケースでは、OpenClawのサブエージェントオーケストレーションを使用して、単一の開発者が14+のAIエージェントを並行実行し、コンテンツ集中型のワークフローのスループットを劇的に向上させる方法を説明します。

## OpenClawの解決策

OpenClawのサブエージェントシステムにより、単一の親エージェントから複数の同時エージェントセッションを生成できます。主な設定オプション：

1. **`sessions_spawn`** - 設定でサブエージェントの生成を有効化
2. **`agents.defaults.subagents.maxConcurrent`** - 最大同時エージェント数を設定（例：10-15）
3. **セッションオーケストレーション** - 親エージェントがすべてのサブエージェントの結果を調整

### ステップバイステップ設定

**1. サブエージェントの生成を有効化**

OpenClaw設定に追加：

```yaml
agents:
  defaults:
    subagents:
      maxConcurrent: 14
sessions_spawn: true
```

**2. エージェントの役割を定義**

ワークフロー内の異なる役割に対して特殊なプロンプトを作成：

- 研究エージェント - 複数のソースから情報を収集
- ライターエージェント - ドラフトコンテンツを作成
- エディターエージェント - コンテンツをレビューして改善
- SEOエージェント - 検索エンジン用に最適化
- 公開エージェント - デプロイを処理

**3. ワークフローを調整**

親エージェントはすべてのサブエージェントに同時にタスクを送信：

```
親: 「トピックXの研究、ドラフトYの作成、Zの最適化を並行して実行」
→ サブエージェント1: トピックXの研究
→ サブエージェント2: ドラフトYの作成
→ サブエージェント3: Zの最適化
親: すべての結果を収集し、最終出力を調整
```

## 結果

Orchestrated AIから文書化されたワークフローに基づく：

- **並行実行**により合計ワークフロー時間が60〜80％削減
- **専門化されたエージェント**は汎用エージェントより高品質な出力を生成
- **スケーラブルなワークフロー** - アーキテクチャを変更せずに異なるタスクに更多のエージェントを追加可能

## 主なポイント

- サブエージェントオーケストレーションは単一エージェントの限界をマルチエージェントのパワーに変換
- 設定は簡単 - `sessions_spawn`を有効にし、同時実行制限を設定するだけ
- 親エージェントは指揮者のように、専門的なサブエージェントを調整
- 研究、コンテンツ作成、多ソース分析など、並列化可能なタスクに最適

## 確認された事実 vs 検証待ち

### ✅ ソースで確認済み

- OpenClaw ショーケースはマルチエージェントオーケストレーションをコア機能として掲載
- `subagents`設定で並行エージェントの生成が可能
- `maxConcurrent`で同時に実行できるエージェント数を制御

### ⚠️ 検証待ち（個別テストが必要）

- 具体的なスループット向上はワークフロータイプによって異なる
- 最適なエージェント数はタスクの複雑さとAPIレート制限に依存

## 実践的なメモ

- 5〜10の同時エージェントから始めて、ニーズに応じて調整
- 各サブエージェントの役割に対して明確で独特なプロンプトを使用
- 親エージェントとサブエージェントの両方でエラー処理を設定
- 並行エージェントはより多くのリソースを消費するため、API使用量を監視
